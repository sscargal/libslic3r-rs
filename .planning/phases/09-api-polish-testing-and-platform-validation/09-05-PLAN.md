  <action>
    Initialize fuzz testing infrastructure using `cargo fuzz`:

    1. Run `cargo fuzz init` in the workspace root to create the `fuzz/` directory structure. If this doesn't work due to workspace setup, create the fuzz directory manually.

    2. Create `fuzz/Cargo.toml`:
    ```toml
    [package]
    name = "slicecore-fuzz"
    version = "0.0.0"
    publish = false
    edition = "2021"

    [package.metadata]
    cargo-fuzz = true

    [dependencies]
    libfuzzer-sys = "0.4"
    slicecore-fileio = { path = "../crates/slicecore-fileio" }

    [workspace]
    members = ["."]

    [[bin]]
    name = "fuzz_stl_binary"
    path = "fuzz_targets/fuzz_stl_binary.rs"
    doc = false

    [[bin]]
    name = "fuzz_stl_ascii"
    path = "fuzz_targets/fuzz_stl_ascii.rs"
    doc = false

    [[bin]]
    name = "fuzz_obj"
    path = "fuzz_targets/fuzz_obj.rs"
    doc = false
    ```

    3. Create `fuzz/fuzz_targets/fuzz_stl_binary.rs`:
    ```rust
    #![no_main]
    use libfuzzer_sys::fuzz_target;

    fuzz_target!(|data: &[u8]| {
        // The parser must never panic on arbitrary input.
        // It should return Ok or Err gracefully.
        let _ = slicecore_fileio::load_mesh(data);
    });
    ```

    4. Create `fuzz/fuzz_targets/fuzz_stl_ascii.rs`:
    ```rust
    #![no_main]
    use libfuzzer_sys::fuzz_target;

    fuzz_target!(|data: &[u8]| {
        // Feed data that starts with "solid " to trigger ASCII STL path.
        // The parser must handle malformed ASCII STL without panic.
        let mut input = b"solid fuzz\n".to_vec();
        input.extend_from_slice(data);
        let _ = slicecore_fileio::load_mesh(&input);
    });
    ```

    5. Create `fuzz/fuzz_targets/fuzz_obj.rs`:
    ```rust
    #![no_main]
    use libfuzzer_sys::fuzz_target;

    fuzz_target!(|data: &[u8]| {
        // Feed data that starts with "v " to trigger OBJ parser path.
        let mut input = b"v 0 0 0\nv 1 0 0\nv 0 1 0\n".to_vec();
        input.extend_from_slice(data);
        let _ = slicecore_fileio::load_mesh(&input);
    });
    ```

    The key invariant: `load_mesh` must NEVER panic on arbitrary input. It should return `Err(...)` for malformed data. If any panic is discovered during fuzz testing, it's a bug to fix.

    Note: `cargo fuzz` requires nightly Rust. The targets are set up to be runnable but we don't need to run an extended fuzz campaign during this plan -- just verify the targets compile and can start. Short smoke test: `cargo +nightly fuzz run fuzz_stl_binary -- -max_total_time=10` (10 seconds).
  </action>

    If nightly is not available: verify `fuzz/Cargo.toml` is valid and targets have correct syntax.

<task type="auto">
  <name>Task 2: Golden file tests for G-code output regression detection</name>
  <files>
    crates/slicecore-engine/tests/golden_tests.rs
  </files>
  <action>
    Create `crates/slicecore-engine/tests/golden_tests.rs` with semantic G-code comparison tests.

    **Approach:** Generate reference G-code on first run, then compare subsequent runs against the reference. Use semantic comparison that ignores comments and whitespace but compares G-code commands and parameters.

    1. Create helper function `assert_gcode_equivalent(actual: &str, expected: &str)`:
       - Split both into lines
       - Filter out comment-only lines (starting with `;`)
       - Filter out empty lines
       - Compare remaining lines, allowing floating-point tolerance (1e-4) for numeric parameters
       - Report first N differences if mismatch

    2. Create helper function `extract_gcode_commands(gcode: &str) -> Vec<String>`:
       - Filter to lines starting with G or M commands
       - Normalize whitespace
       - Return sorted or ordered command list

    3. Create synthetic mesh builder functions (same as benchmarks -- reuse the calibration cube, cylinder patterns):
       - `build_golden_cube()` -- 20mm cube centered at (100, 100)
       - `build_golden_cylinder()` -- 10mm diameter, 20mm tall cylinder

    4. Create golden tests:
       - `golden_calibration_cube_default`: Slice cube with default config, compare against embedded reference.
         The first time, generate the reference by slicing and storing the output. For determinism, verify that two consecutive slices produce identical output. Store the reference as a const string or use `include_str!` from a golden file.
       - `golden_calibration_cube_fine`: Slice cube with 0.1mm layer height, compare against reference.
       - `golden_cylinder_default`: Slice cylinder with default config.

    5. Since the pipeline is deterministic (verified in Phase 3), the golden approach works:
       - Generate reference G-code, store as `tests/golden/` files or as `include_str!` embedded strings
       - On each test run, slice the same model+config and compare
       - If the golden file doesn't exist, create it (first-run behavior) -- but for CI, golden files must be committed

    **Simpler alternative (preferred for automation):** Instead of external golden files, embed the expected structural characteristics:
    - Expected layer count
    - Expected G-code line count (within +/-5%)
    - First N and last N G-code commands match
    - Specific commands present (M104, M109, G28, G0/G1 ratios, etc.)
    - Total E-axis value within tolerance
    - This is more maintainable than byte-exact golden files

    Use this structural comparison approach. Create tests that verify:
    - Layer count matches expected
    - G-code starts with correct preamble (G28, M104/M109, etc.)
    - G-code ends with correct postamble (M104 S0, M140 S0, etc.)
    - Feature type comments present (;TYPE:)
    - Total extrusion (final E value) within 5% of expected
    - Determinism: two slices produce identical output
  </action>
  <verify>
    Run `cargo test -p slicecore-engine --test golden_tests` passes.
    At least 3 golden tests run and verify G-code structure.
    Determinism test confirms identical output from identical input.
  </verify>
  <done>
    Golden file tests verify G-code structural correctness for multiple model+config combinations. Determinism is verified. Tests detect regressions in layer count, preamble/postamble, feature types, and extrusion totals.
  </done>
