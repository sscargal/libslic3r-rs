<objective>
Enrich all firmware dialects with dialect-specific inline commands (pressure advance, acceleration, jerk/junction) and make the G-code dialect configurable in the engine pipeline.

Purpose: The engine currently hardcodes Marlin dialect. Users targeting Klipper, RepRapFirmware, or Bambu printers need dialect-specific commands emitted throughout the print body, not just in start/end sequences. Acceleration and jerk control (GCODE-06) must emit the correct commands per dialect.

Output: Configurable dialect in PrintConfig, extended GcodeCommand enum, dialect-specific acceleration/jerk/pressure-advance formatting, enriched Klipper/RepRap/Bambu dialect modules.
</objective>

  <name>Task 1: Extend GcodeCommand enum and dialect modules with acceleration/jerk/pressure-advance commands</name>
  <files>
    crates/slicecore-gcode-io/src/commands.rs
    crates/slicecore-gcode-io/src/dialect.rs

    crates/slicecore-gcode-io/src/marlin.rs
    crates/slicecore-gcode-io/src/validate.rs
  </files>
  <action>
1. In commands.rs, add these new variants to GcodeCommand:
   - `ArcMoveCW { x: Option<f64>, y: Option<f64>, i: f64, j: f64, e: Option<f64>, f: Option<f64> }` -- G2 clockwise arc
   - `ArcMoveCCW { x: Option<f64>, y: Option<f64>, i: f64, j: f64, e: Option<f64>, f: Option<f64> }` -- G3 counter-clockwise arc
   - `SetAcceleration { print_accel: f64, travel_accel: f64 }` -- M204 P{print} T{travel}
   - `SetJerk { x: f64, y: f64, z: f64 }` -- M205 X{x} Y{y} Z{z}
   - `SetPressureAdvance { value: f64 }` -- dialect-dependent formatting
   - `ToolChange(u8)` -- T{n} tool change
   Add Display impl for each. For SetAcceleration use M204 P/T. For SetJerk use M205 X/Y/Z. For SetPressureAdvance use M900 K{value} (Marlin default). For ArcMoveCW/CCW format as G2/G3 X Y I J [E] [F] using same decimal precision as LinearMove (X/Y 3 decimals, E 5 decimals, F 1 decimal). For ToolChange format as T{n}.

2. In dialect.rs, add a `DialectCommands` trait or helper functions:
   - `format_acceleration(dialect: GcodeDialect, print_accel: f64, travel_accel: f64) -> String` that returns:
     - Marlin: `M204 P{print} T{travel}`
     - Klipper: `SET_VELOCITY_LIMIT ACCEL={print}`
     - RepRapFirmware: `M204 S{print}` (single value, not P/T split)
     - Bambu: `M204 P{print} T{travel}` (same as Marlin)
   - `format_pressure_advance(dialect: GcodeDialect, value: f64) -> String` that returns:
     - Marlin: `M900 K{value}`
     - Klipper: `SET_PRESSURE_ADVANCE ADVANCE={value}`
     - RepRapFirmware: `M572 D0 S{value}`
     - Bambu: `M900 K{value}` (same as Marlin)
   - `format_jerk(dialect: GcodeDialect, x: f64, y: f64, z: f64) -> String`:
     - Marlin: `M205 X{x} Y{y} Z{z}`
     - Klipper: `SET_VELOCITY_LIMIT SQUARE_CORNER_VELOCITY={x}`
     - RepRapFirmware: `M566 X{x*60} Y{y*60} Z{z*60}` (RepRap uses mm/min for jerk)
     - Bambu: `M205 X{x} Y{y} Z{z}` (same as Marlin)
   Add the GcodeDialect import if needed from dialect.rs.
   Note: The Display impl for SetAcceleration/SetJerk/SetPressureAdvance should use the Marlin format as default; the dialect-aware formatting functions provide dialect-specific strings for use via GcodeCommand::Raw when needed. Alternatively, consider making SetAcceleration/SetJerk/SetPressureAdvance produce the Marlin-compatible format in Display, and having the engine use the dialect formatting functions to produce Raw commands for non-Marlin dialects.

3. Enrich Klipper start_gcode in klipper.rs: no changes needed beyond existing (already has BED_MESH_CALIBRATE and TURN_OFF_HEATERS). The dialect-specific inline commands (SET_PRESSURE_ADVANCE, SET_VELOCITY_LIMIT) are emitted by the engine via the formatting functions above.

4. Enrich RepRap module reprap.rs: add any RepRap-specific start sequence adjustments (M572 for pressure advance in start if desired). The existing start/end sequence should be reviewed to ensure it uses M204 S instead of P/T if any acceleration commands are present.

5. Enrich Bambu module bambu.rs: add M620/M621 AMS filament slot commands to start sequence as comments noting they're used for AMS tool changes. Add a note that full AMS support requires 3MF format.

6. In validate.rs, extend validation to accept:
   - G2 and G3 commands (arc moves) with I/J parameters
   - T commands (tool changes) as valid line prefixes
   - SET_PRESSURE_ADVANCE, SET_VELOCITY_LIMIT, M572, M566 as valid extended commands
   - M620, M621 as valid Bambu-specific commands

7. Add tests in each modified file for the new command formatting and validation.
  </action>
  <verify>
`cargo test -p slicecore-gcode-io` -- all tests pass including new ones for arc commands, acceleration formatting, jerk formatting, pressure advance formatting, and extended validation.
  </verify>
  <done>
GcodeCommand enum has ArcMoveCW, ArcMoveCCW, SetAcceleration, SetJerk, SetPressureAdvance, ToolChange variants. Each dialect has formatting functions for acceleration/jerk/pressure-advance. Validator accepts G2, G3, T, and dialect-specific extended commands. All tests pass.
  </done>
