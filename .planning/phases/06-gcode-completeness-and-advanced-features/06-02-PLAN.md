Implement per-feature flow control (ADV-05), custom G-code injection hooks (ADV-04), and ironing for top surface quality (ADV-08).

Purpose: These three features enhance print quality and user control. Per-feature flow control lets users fine-tune extrusion for different feature types. Custom G-code injection enables firmware-specific macros and user workflows. Ironing provides smooth top surfaces -- a standard feature in all major slicers.

Output: Per-feature flow multiplier system, custom G-code hook framework, ironing pass generator with FeatureType::Ironing variant.
</objective>

  <action>
1. Create flow_control.rs with:
   - `PerFeatureFlow` struct containing a `HashMap<String, f64>` (or parallel struct with named fields for each FeatureType) mapping each FeatureType to a flow multiplier (default 1.0 for all).
   - Fields for: outer_perimeter, inner_perimeter, solid_infill, sparse_infill, support, support_interface, bridge, gap_fill, skirt, brim, variable_width_perimeter, ironing (anticipate new variant). All default to 1.0.
   - Implement Default, Serialize, Deserialize with `#[serde(default)]`.
   - `get_multiplier(&self, feature: FeatureType) -> f64` method that matches FeatureType to the corresponding field.
   - Add `per_feature_flow: PerFeatureFlow` field to PrintConfig in config.rs with default.

2. Create custom_gcode.rs with:
   - `CustomGcodeHooks` struct with fields for each injection point:
     - `before_layer_change: String` -- injected before each layer's Z move
     - `after_layer_change: String` -- injected after each layer's Z move
     - `tool_change_gcode: String` -- injected during tool changes (future use)
     - `before_every_layer: String` -- alias/synonym
     - `custom_gcode_per_z: Vec<(f64, String)>` -- (z_height, gcode) pairs for specific Z heights
   - Implement Default (all empty strings, empty vec), Serialize, Deserialize.
   - `substitute_placeholders(gcode: &str, layer_num: usize, layer_z: f64, ...) -> String` function that replaces `{layer_num}`, `{layer_z}`, `{total_layers}` with actual values.
   - Add `custom_gcode: CustomGcodeHooks` field to PrintConfig.

3. Add `FeatureType::Ironing` variant to toolpath.rs FeatureType enum.
   - Also add `FeatureType::PurgeTower` variant (needed for Plan 07 multi-material, but define the variant now to avoid conflicts).

4. In gcode_gen.rs, modify `generate_layer_gcode` or `generate_full_gcode`:
   - Apply per-feature flow: multiply each segment's e_value by the flow multiplier for its feature type when assembling G-code. Do this in the extrusion move emission path: `e_value * per_feature_flow.get_multiplier(seg.feature)`.
   - Inject custom G-code: before each layer's Z move, emit before_layer_change (if non-empty). After Z move, emit after_layer_change. Check custom_gcode_per_z for matching Z heights (within 0.001mm tolerance).
   - Add `feature_label` mapping for Ironing and PurgeTower variants.

5. Add `pub mod flow_control;` and `pub mod custom_gcode;` to lib.rs.

6. Tests:
   - PerFeatureFlow default has all 1.0 multipliers.
   - Setting outer_perimeter to 0.95 reduces E-value for outer perimeter segments.
   - Custom gcode placeholder substitution replaces {layer_num} and {layer_z}.
   - TOML deserialization of per_feature_flow and custom_gcode sections.
  </action>
  <verify>
`cargo test -p slicecore-engine` -- all tests pass including new flow control and custom gcode tests.
  </verify>

1. Create ironing.rs with:
   - `IroningConfig` struct with fields:
     - `enabled: bool` (default false)
     - `flow_rate: f64` (default 0.1 -- 10% flow)
     - `speed: f64` (default 15.0 mm/s)
     - `spacing: f64` (default 0.1 mm -- very tight line spacing)
     - `angle: f64` (default 45.0 degrees -- offset from primary infill)
   - Implement Default, Serialize, Deserialize with `#[serde(default)]`.
   - `generate_ironing_passes(top_regions: &[ValidPolygon], config: &IroningConfig, layer_z: f64, nozzle_diameter: f64, layer_height: f64, filament_diameter: f64, extrusion_multiplier: f64) -> Vec<ToolpathSegment>` function that:
     - Uses rectilinear infill generation internally with 100% density and the configured spacing
     - Converts the resulting infill lines to ToolpathSegment with FeatureType::Ironing
     - Applies ironing flow_rate to E-values (very low flow, just enough to smooth)
     - Uses ironing speed for feedrate
     - The angle should be configurable but defaults to primary_infill_angle + 45 degrees

2. Add `ironing: IroningConfig` field to PrintConfig in config.rs with default.

3. In engine.rs, integrate ironing into the per-layer pipeline:
   - After surface classification identifies top surfaces, if `config.ironing.enabled` is true, generate ironing passes for top_surface regions.
   - Append ironing toolpath segments to the layer toolpath AFTER all other features (perimeters, infill, support).
   - Ironing only on layers identified as having top surfaces (top_solid_layers classification).

4. Add `pub mod ironing;` to lib.rs.

5. Tests:
   - IroningConfig default has correct values (enabled=false, flow_rate=0.1, speed=15.0).
   - generate_ironing_passes with a simple square polygon produces segments with FeatureType::Ironing.
   - E-values in ironing segments are significantly lower than normal extrusion segments (multiplied by flow_rate).
   - Ironing segments use the configured speed.
   - TOML deserialization of ironing config section.
