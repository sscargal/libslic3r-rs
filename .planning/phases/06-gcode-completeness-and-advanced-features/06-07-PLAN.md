1. In config.rs, define:
   - `ToolConfig` struct: `nozzle_temp: f64`, `retract_length: f64`, `retract_speed: f64`. Derive Debug, Clone, Default, Serialize, Deserialize.
   - `MultiMaterialConfig` struct:
     - `enabled: bool` (default false)
     - `tool_count: u8` (default 1)
     - `tools: Vec<ToolConfig>` (default empty)
     - `purge_tower_position: [f64; 2]` (default [200.0, 200.0])
     - `purge_tower_width: f64` (default 15.0 mm)
     - `purge_volume: f64` (default 70.0 mm^3)
     - `wipe_length: f64` (default 2.0 mm)
     Derive Debug, Clone, Default, Serialize, Deserialize with `#[serde(default)]`.
   - Add `multi_material: MultiMaterialConfig` to PrintConfig.

2. Create multimaterial.rs with:
   - `ToolChangeSequence` struct: `commands: Vec<GcodeCommand>` -- the full tool change sequence
   - `generate_tool_change(from_tool: u8, to_tool: u8, config: &MultiMaterialConfig, print_config: &PrintConfig) -> ToolChangeSequence` that produces:
     - Retract current tool
     - Move to purge tower position (travel)
     - Emit ToolChange(to_tool) command (T-code)
     - Prime new tool (unretract + purge extrusion)
     - Wipe move across purge tower
     - Return the command sequence

   - `PurgeTowerLayer` struct: `commands: Vec<GcodeCommand>`, `is_dense: bool`
   - `generate_purge_tower_layer(layer_z: f64, layer_height: f64, config: &MultiMaterialConfig, has_tool_change: bool, nozzle_diameter: f64) -> PurgeTowerLayer`:
     - If has_tool_change: generate dense perimeters + infill filling the purge tower area (the actual purge)
     - If no tool change: generate sparse infill (2 perimeters only) to maintain tower structural integrity
     - Use FeatureType::PurgeTower for all segments
     - The purge tower is a simple rectangular region at the configured position with configured width

   - `assign_tools_per_region(contours: &[Vec<ValidPolygon>], modifier_tools: &[(ModifierMesh, u8)]) -> Vec<u8>`:
     - Simple tool assignment: for each contour region, check if it falls within a modifier mesh assigned to a specific tool
     - Returns tool index per region (default tool 0 for unassigned regions)

3. In engine.rs, add a `slice_multi_material` method or integrate into existing flow:
   - When multi_material.enabled is true:
     - Track active tool across layers
     - When tool changes are needed (different regions need different tools), insert tool change sequences
     - Generate purge tower on every layer
     - Insert purge tower toolpath at the end of each layer

4. Add `pub mod multimaterial;` to lib.rs.

5. Tests:
   - generate_tool_change produces ToolChange command with correct tool number
   - Tool change sequence includes retract, travel to purge tower, T-code, prime
   - generate_purge_tower_layer with tool change produces dense infill
   - generate_purge_tower_layer without tool change produces sparse perimeters only
   - MultiMaterialConfig default has enabled=false
   - TOML deserialization of multi_material section
  </action>

<task type="auto">
  <name>Task 2: Sequential printing with collision detection</name>
  <files>
    crates/slicecore-engine/src/sequential.rs
    crates/slicecore-engine/src/config.rs
    crates/slicecore-engine/src/engine.rs
    crates/slicecore-engine/src/lib.rs
  </files>
  <action>
1. In config.rs, define:
   - `SequentialConfig` struct:
     - `enabled: bool` (default false)
     - `extruder_clearance_radius: f64` (default 35.0 mm -- typical print head radius)
     - `extruder_clearance_height: f64` (default 40.0 mm -- height above nozzle to carriage)
     Derive Debug, Clone, Default, Serialize, Deserialize with `#[serde(default)]`.
   - Add `sequential: SequentialConfig` to PrintConfig.

2. Create sequential.rs with:
   - `ObjectBounds` struct:
     - `min_x: f64, max_x: f64, min_y: f64, max_y: f64, max_z: f64`
     - `object_index: usize`

   - `detect_collision(obj_a: &ObjectBounds, obj_b: &ObjectBounds, clearance_radius: f64, clearance_height: f64) -> bool`:
     - If both objects are shorter than clearance_height, only check XY clearance
     - If either object is taller than clearance_height, the carriage can collide -- check if XY distance between bounding boxes is less than clearance_radius
     - XY distance = gap between bounding boxes (not center-to-center)
     - Return true if collision detected

   - `order_objects(objects: &[ObjectBounds], clearance_radius: f64, clearance_height: f64) -> Result<Vec<usize>, String>`:
     - Sort by max_z (shortest first -- minimizes collision risk)
     - For each pair of consecutive objects in the order, check for collisions
     - If collision detected, try alternative orderings or return error
     - Return ordered indices

   - `plan_sequential_print(meshes: &[(TriangleMesh, ObjectBounds)], config: &PrintConfig) -> Result<Vec<(usize, Vec<u8>)>, EngineError>`:
     - Slice each mesh independently
     - Generate full G-code for each object
     - Insert inter-object travel to safe Z (above clearance_height)
     - Return ordered list of (object_index, gcode_bytes)

3. In engine.rs, add a `slice_sequential` method:
   - Takes multiple meshes
   - Checks collision feasibility
   - Slices each independently
   - Concatenates G-code with inter-object transitions
   - Returns combined SliceResult

4. Add `pub mod sequential;` to lib.rs.

5. Tests:
   - Two small objects far apart: no collision detected, valid ordering
   - Two objects too close together with one taller than clearance_height: collision detected
   - order_objects sorts shortest first
   - SequentialConfig default has enabled=false
   - TOML parsing of sequential config section
  </action>
  <verify>
`cargo test -p slicecore-engine` -- all tests pass.
`cargo clippy -p slicecore-engine -- -D warnings` -- no warnings.
  </verify>
  <done>
Sequential printing slices objects independently, detects collisions using extruder clearance envelope, orders objects shortest-first, and concatenates G-code with safe Z transitions. Disabled by default.
  </done>
