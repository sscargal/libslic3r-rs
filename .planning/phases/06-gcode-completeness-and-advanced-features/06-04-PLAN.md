<objective>
Implement G2/G3 arc fitting as a post-processing step that converts sequences of linear moves into circular arc commands, reducing G-code file size while maintaining dimensional accuracy.

Purpose: Arc fitting (GCODE-11) is a key optimization that reduces G-code file size by 20-40% on curved models. This reduces SD card read time, network transfer time, and buffer overflow risk on older firmware. The algorithm runs as a post-processing pass on the final GcodeCommand stream.

Output: Arc fitting module in slicecore-gcode-io, integrated as optional post-processing in the engine pipeline.
</objective>

<task type="auto">
  <name>Task 1: Arc fitting algorithm</name>
  <files>
    crates/slicecore-gcode-io/src/arc.rs
    crates/slicecore-gcode-io/src/lib.rs
  </files>
  <action>
1. Create arc.rs with the arc fitting algorithm:

   - `circumcircle(p1: (f64, f64), p2: (f64, f64), p3: (f64, f64)) -> Option<((f64, f64), f64)>` that computes the circumscribed circle center and radius from 3 points. Returns None if points are collinear (determinant < 1e-10).

   - `points_fit_arc(points: &[(f64, f64)], tolerance: f64) -> Option<((f64, f64), f64, bool)>` that:
     - Takes a slice of 2D points (at least 3)
     - Computes circumcircle from first, middle, and last points
     - Checks all intermediate points lie within `tolerance` of the arc radius
     - Determines arc direction (CW or CCW) via cross product
     - Returns (center, radius, is_ccw) or None

   - `fit_arcs(commands: &[GcodeCommand], tolerance: f64, min_arc_points: usize) -> Vec<GcodeCommand>` that:
     - Scans the command stream for consecutive G1 (LinearMove) sequences
     - Uses a sliding window approach: start with min_arc_points moves, try to extend
     - For each candidate window, extract XY positions, test with points_fit_arc
     - If arc fits, compute I/J offsets (center minus start point), sum E-values
     - Replace the window with a single ArcMoveCW or ArcMoveCCW command
     - Preserve non-G1 commands (comments, retractions, Z-moves, etc.) as-is -- they break the arc sequence
     - Skip arcs with radius < 0.5mm (too tight) or > 1000mm (nearly straight)
     - E-value for arc = sum of replaced segment E-values (acceptable since deviation is within tolerance)
     - Feedrate for arc = feedrate of the last replaced segment

   - `arc_length(center: (f64, f64), start: (f64, f64), end: (f64, f64), is_ccw: bool) -> f64` helper to compute the arc's angular span and length for potential E redistribution.

2. Add `pub mod arc;` to lib.rs and re-export `fit_arcs` in the public API.

3. Tests:
   - circumcircle of 3 points on a known circle returns correct center and radius
   - circumcircle of 3 collinear points returns None
   - points_fit_arc with points on a circle succeeds
   - points_fit_arc with one outlier beyond tolerance returns None
   - fit_arcs on a sequence of G1 moves forming a semicircle produces G2/G3 commands
   - fit_arcs preserves non-G1 commands (comments, retractions)
   - fit_arcs preserves total E-value across replaced segments
   - fit_arcs with no arcs (all straight lines) returns original commands unchanged
   - fit_arcs skips sequences shorter than min_arc_points
  </action>
  <verify>
`cargo test -p slicecore-gcode-io` -- all tests pass including arc fitting tests.
  </verify>
  <done>
Arc fitting algorithm correctly identifies circular arc patterns in G1 sequences, replaces them with G2/G3 commands, preserves E-values, and handles edge cases (collinear, too short, too tight radius).
  </done>
</task>
