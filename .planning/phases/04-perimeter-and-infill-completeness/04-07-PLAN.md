  <action>
    Implement adaptive cubic infill in `infill/adaptive_cubic.rs`.

    **Concept:** Like cubic infill but with variable density. Near the outer surface of the object, the infill is dense (small cube cells). Deep inside, it's sparse (large cube cells). This is achieved with an octree that subdivides cells near surface proximity.

    **Algorithm:**

    1. **Build 2D quadtree for the layer** (simplified from 3D octree for per-layer generation):
       - Compute the bounding box of the infill region.
       - Start with a single cell covering the entire bbox.
       - Recursively subdivide cells that are near the infill region boundary.
       - "Near boundary" = cell center is within `threshold` mm of any polygon edge (where threshold = spacing * subdivision_factor).
       - Max depth = 4-6 levels (configurable). Depth 0 = single cell, depth 6 = 64 cells per axis.
       - Stop subdividing if cell size < line_width (too small to print).

    2. **Generate cubic lines within each cell:**
       - For each leaf cell of the quadtree, generate parallel lines at the cell's density.
       - Use the same 3-angle cycling as cubic infill (0/60/120 degrees based on layer_index).
       - Cell spacing = cell_size (larger cells = sparser infill, smaller cells = denser).

    3. **Clip lines to infill region:**
       - Use the same scanline clipping as rectilinear.

    **Simplified approach for Phase 4:**
    Since full 3D octree is complex, use a 2D approach:
    - Compute distance from each grid point to the nearest polygon edge of the infill_region boundary.
    - Use this distance to determine local spacing: `local_spacing = base_spacing * (1.0 + distance / max_distance * density_reduction_factor)`.
    - Near edges (distance ~ 0): spacing = base_spacing (dense).
    - Far from edges (distance ~ max): spacing = base_spacing * density_reduction_factor (sparse).
    - Generate lines with varying spacing and clip.

    Even simpler (recommended for first implementation):
    - Use the quadtree approach with distance-to-boundary for subdivision.
    - Generate cubic-pattern lines in each leaf cell.

    ```rust
    pub fn generate(
        infill_region: &[ValidPolygon],
        density: f64,
        layer_index: usize,
        layer_z: f64,
        line_width: f64,
    ) -> Vec<InfillLine>
    ```

    **Quadtree struct (internal):**
    ```rust
    struct QuadCell {
        x_min: f64, y_min: f64, x_max: f64, y_max: f64,
        depth: u32,
        children: Option<Box<[QuadCell; 4]>>,
    }
    ```

    **Tests:**
    - 20mm square: produces infill lines.
    - Lines near the edges of the region are more densely spaced than lines in the center.
    - Total infill line count is between pure-dense and pure-sparse (adaptive behavior).
    - Lines are within bounding box.
    - Different layer indices produce different angles (cubic cycling).
    - Empty region returns empty.

    Wire dispatch: `InfillPattern::AdaptiveCubic => adaptive_cubic::generate(...)`.

  <action>
    Implement lightning infill in `infill/lightning.rs`.

    **Concept:** Lightning infill is NOT traditional infill -- it generates minimal tree-branching support structures that only exist where needed to hold up top surfaces. Branches grow downward from top surface regions, merging when close to each other, creating a lightning-bolt-like pattern.

    **Algorithm:**

    Lightning requires cross-layer awareness. Unlike other patterns that are generated per-layer independently, lightning needs a pre-pass that analyzes all layers to determine where branches should go.

    **LightningContext (cross-layer state):**
    ```rust
    pub struct LightningContext {
        /// Branch trees indexed by layer, growing downward from top surfaces.
        pub layer_branches: Vec<Vec<LightningBranch>>,
    }

    struct LightningBranch {
        start: IPoint2,   // Upper point (closer to top surface)
        end: IPoint2,     // Lower point (where branch grows to)
        width: f64,       // Extrusion width
    }
    ```

    **Pre-pass (called once before per-layer generation):**
    ```rust
    pub fn build_lightning_context(
        layers: &[(Vec<ValidPolygon>, SurfaceClassification)], // (contours, surfaces) per layer
        density: f64,
        line_width: f64,
    ) -> LightningContext
    ```

    1. **Identify support requirements:** For each layer, determine which regions need support from below. Top surface regions (identified by SurfaceClassification) need infill support within `density * line_width` spacing.

    2. **Grow branches downward:** Starting from the topmost layer with top surfaces:
       a. Sample support points on top surface regions at spacing determined by density.
       b. Each support point becomes a branch root.
       c. Grow each branch one layer downward (straight down initially).
       d. At each lower layer:
          - Check if the branch point is within the infill region for that layer.
          - If two branches are within `merge_distance` (2 * line_width) of each other, merge them (one branch continues, the other terminates).
          - Branches can spread laterally at a maximum angle (45 degrees from vertical).
       e. Continue until the branch reaches a bottom surface or the build plate.

    3. **Per-layer output:** For each layer, collect all branch segments that pass through that layer.

    **Simplified implementation for Phase 4:**
    Full tree merging is complex. Simplified approach:
    - For each top surface region, generate support points at regular spacing.
    - Each point becomes a vertical column extending downward through all layers until it hits a bottom surface or the bed.
    - At each layer, the column intersection is a point. Connect adjacent column points with horizontal segments to form a sparse network.
    - Merge nearby columns (within merge_distance) into one.
    - Output: for each layer, the horizontal connection segments between columns + short vertical segments.

    This simplified version produces less optimal results than full tree merging but is functionally correct and much simpler to implement.

    ```rust
    pub fn generate(
        infill_region: &[ValidPolygon],
        density: f64,
        layer_index: usize,
        line_width: f64,
        context: Option<&LightningContext>,
    ) -> Vec<InfillLine>
    ```

    **Engine integration:**
    The engine needs to build the LightningContext before per-layer processing when lightning infill is selected:
    ```rust
    let lightning_ctx = if self.config.infill_pattern == InfillPattern::Lightning {
        Some(build_lightning_context(&layer_data, self.config.infill_density, extrusion_width))
    } else {
        None
    };
    ```
    Then pass `lightning_ctx.as_ref()` to `generate_infill()`.

    **Update generate_infill dispatch:**
    Lightning needs special handling since it takes an optional context:
    ```rust
    InfillPattern::Lightning => {
        lightning::generate(infill_region, density, layer_index, line_width, lightning_context)
    }
    ```
    Add `lightning_context: Option<&lightning::LightningContext>` parameter to `generate_infill()`.

    **Tests:**
    - Lightning on a flat-top box: produces sparse vertical supports under the top surface.
    - Lightning uses fewer total line segments than rectilinear at the same density.
    - No infill lines in regions with no top surface above them.
    - Empty infill region returns empty.
    - Context build doesn't panic on empty layers.

    Wire dispatch: `InfillPattern::Lightning => lightning::generate(...)`.
