1. Add dependencies to `crates/slicecore-engine/Cargo.toml`:
```toml
slicecore-config-schema = { path = "../slicecore-config-schema" }
slicecore-config-derive = { path = "../slicecore-config-derive" }
```

2. Add imports to config.rs:
```rust
use slicecore_config_derive::SettingSchema;
use slicecore_config_schema::HasSettingSchema;
```

3. Annotate ALL enums in config.rs with `#[derive(SettingSchema)]` and variant attributes:

For each enum, add `#[derive(SettingSchema)]` to the derive list and `#[setting(display = "...", description = "...")]` to each variant. The display name is the human-readable label; description explains what the variant does.

Enums to annotate (all in config.rs):
- WallOrder (2 variants: InnerFirst, OuterFirst)
- SurfacePattern (6 variants: Rectilinear, Monotonic, MonotonicLine, Concentric, Hilbert, Archimedean)
- BedType (6 variants)
- InternalBridgeMode (3 variants)
- BrimType (4 variants)
- CustomGcodeTrigger (~5 variants)
- SlicingTolerance (~3 variants)
- ScarfJointType (~3 variants)

For each variant, use `#[setting(display = "Human Readable Name", description = "What this option does")]`.

4. Annotate the first group of structs (approximately the first half of config.rs). For each struct:

a. Add `#[derive(SettingSchema)]` to the derive attribute list (alongside existing Debug, Clone, Serialize, Deserialize).

b. Add struct-level `#[setting(category = "CategoryName")]` where CategoryName matches the SettingCategory enum variant.

c. For EVERY field, add `#[setting(...)]` with:
   - `tier = N` (from TIER_MAP.md)
   - `description = "..."` (based on existing doc comments, but as explicit setting attribute)
   - `units = "..."` where applicable (mm, mm/s, deg_c, %, etc.)
   - `min = X.X, max = Y.Y` for numeric fields with valid ranges
   - `affects = [...]` for fields that influence other settings
   - `depends_on = "..."` for fields that only apply when another setting is enabled
   - `tags = [...]` for searchable categorization

d. For sub-struct fields (e.g., `pub speed: SpeedConfig`), use `#[setting(flatten)]`.

e. For `passthrough: BTreeMap<String, String>`, use `#[setting(skip)]`.

Structs to annotate in this task (first half):
- LineWidthConfig (7 fields, category = "LineWidth")
- SpeedConfig (~21 fields, category = "Speed")
- CoolingConfig (~15 fields, category = "Cooling")
- RetractionConfig (~12 fields, category = "Retraction")
- MachineConfig (~60 fields, category = "Machine")
- FuzzySkinConfig (3 fields, category = "Advanced")
- BrimSkirtConfig (~8 fields, category = "Adhesion")
- InputShapingConfig (~5 fields, category = "Advanced")
- ToolChangeRetractionConfig (2 fields, category = "Retraction")
- DimensionalCompensationConfig (3 fields, category = "Advanced")
- AccelerationConfig (~25 fields, category = "Acceleration")

Use existing doc comments as the basis for description text. For example:
- Doc: `/// Perimeter print speed (mm/s).` -> `#[setting(description = "Perimeter print speed", units = "mm/s")]`
- Doc: `/// Number of bottom solid layers.` -> `#[setting(description = "Number of bottom solid layers")]`

For `affects` relationships, use 3D printing domain knowledge:
- Speed fields affect: print time, quality
- Temperature fields affect: adhesion, quality, stringing
- Retraction fields affect: stringing, oozing
- Layer height affects: almost everything
- Cooling fields affect: bridging, overhangs, quality

The `depends_on` field uses simple key references:
- Support sub-settings: `depends_on = "support.enable"`
- Tree support settings: `depends_on = "support.support_type == Tree"`
- Brim settings: `depends_on = "brim.enable"`
