<task type="auto">
  <name>Task 1: Create calibration cube test fixture and G-code structure tests</name>
  <files>
    crates/slicecore-engine/tests/calibration_cube.rs
    crates/slicecore-engine/tests/integration.rs
  </files>
  <action>
    Create integration tests that verify the full pipeline with a synthetic 20mm calibration cube.

    **Test fixture -- 20mm calibration cube mesh:**
    Create a helper function `fn calibration_cube_20mm() -> TriangleMesh` that builds a 20mm x 20mm x 20mm cube mesh programmatically:
    - 8 vertices: all combinations of (0,0,0) to (20,20,20).
    - 12 triangles (2 per face, 6 faces).
    - Use the same winding convention as the unit_cube in slicecore-mesh tests, just scaled 20x.
    - Optionally center on bed: translate to (100, 100, 0) so it's roughly centered on a 220x220 bed.

    **calibration_cube.rs tests:**

    1. `test_calibration_cube_produces_gcode`:
       - Slice the 20mm cube with default PrintConfig.
       - Verify result.gcode is non-empty.
       - Verify result.layer_count is approximately 20mm / 0.2mm = ~100 layers (within 5 layers tolerance for first layer height difference).
       - Verify G-code string contains expected commands: G28, M83, M104, M109, M140, M190, G1, M107.

    2. `test_gcode_has_start_and_end_sequences`:
       - Parse the G-code output as a string.
       - Verify it starts with homing (G28) within the first 20 lines.
       - Verify it ends with fan off (M107) and steppers disabled (M84) in the last 10 lines.

    3. `test_gcode_has_temperature_commands`:
       - G-code contains M109 (wait for nozzle temp) and M190 (wait for bed temp).
       - First layer temps match first_layer_nozzle_temp and first_layer_bed_temp.

    4. `test_gcode_has_retraction`:
       - G-code contains retract patterns: `G1 E-{distance}` somewhere in the output.

    5. `test_gcode_has_fan_commands`:
       - G-code contains M106 (fan on) after disable_fan_first_layers.
       - G-code does NOT contain M106 before the first layer (or within the start sequence region).

    **integration.rs tests:**

    6. `test_gcode_passes_validation`:
       - Slice the cube, get G-code string.
       - Run `slicecore_gcode_io::validate_gcode()` on the output.
       - Assert validation passes (no errors; warnings are acceptable).

    7. `test_infill_density_zero_and_hundred`:
       - Slice with infill_density = 0.0. Should produce G-code (perimeters only, no infill between walls).
       - Slice with infill_density = 1.0. Should produce G-code with more extrusion moves (solid fill).
       - Compare: 100% infill G-code file size should be significantly larger than 0% (at least 2x).

    8. `test_skirt_present_in_output`:
       - Slice with skirt_loops = 1.
       - Verify G-code output contains skirt-related moves (extrusion moves before the first perimeter on layer 0). Use a heuristic: the first extrusion moves at Z=first_layer_height should be the skirt.

    9. `test_brim_works`:
       - Slice with brim_width = 5.0.
       - Verify G-code output is larger than without brim (more first-layer extrusion).
  </action>
  <verify>
    `cargo test -p slicecore-engine --test calibration_cube` -- all tests pass.
    `cargo test -p slicecore-engine --test integration` -- all tests pass.
  </verify>
  <done>
    Calibration cube G-code has correct structure (start sequences, temperature, retraction, fan, end sequences). G-code passes validation. Infill density and skirt/brim are configurable.
  </done>
</task>

<task type="auto">
  <name>Task 2: Determinism test and layer height variation test</name>
  <files>
    crates/slicecore-engine/tests/determinism.rs
  </files>
  <action>
    Create tests verifying determinism and layer height configurability.

    **determinism.rs tests:**

    1. `test_deterministic_output`:
       - Create a 20mm calibration cube mesh.
       - Create a PrintConfig::default().
       - Call Engine::slice() twice with the exact same inputs.
       - Assert the two G-code outputs are byte-for-byte identical: `assert_eq!(result1.gcode, result2.gcode)`.
       - This directly verifies Phase 3 Success Criterion 3.

    2. `test_deterministic_with_custom_config`:
       - Same as above but with a non-default config (layer_height=0.15, infill_density=0.3, wall_count=3).
       - Verify byte-for-byte identical output across two runs.

    3. `test_layer_height_variation`:
       - Slice the cube with layer_height = 0.2 (default). Record layer_count_02.
       - Slice the same cube with layer_height = 0.1. Record layer_count_01.
       - Assert: `layer_count_01` is approximately `2 * layer_count_02` (within 10% tolerance for first-layer height effects and rounding).
       - Both outputs should be non-empty, valid G-code.
       - This directly verifies Phase 3 Success Criterion 4.

    4. `test_layer_height_produces_valid_gcode_both`:
       - Validate G-code output for both 0.2mm and 0.1mm layer heights.
       - Both must pass `validate_gcode()`.

    5. `test_different_configs_produce_different_output`:
       - Slice with two different configs (e.g., different layer heights).
       - Assert the outputs are NOT identical (sanity check that config actually affects output).

    **Phase 3 SC verification summary** (as comments in the test file):

    - SC2: CLI accepts correct arguments (tested in 03-05 plan build + help text)
    - SC3: Deterministic output (test_deterministic_output)
    - SC4: Layer height variation (test_layer_height_variation)
    - SC5: Skirt/brim and infill density (tested in integration.rs)
  </action>
