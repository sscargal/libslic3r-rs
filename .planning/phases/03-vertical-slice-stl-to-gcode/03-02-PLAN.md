    Implement perimeter shell generation in `perimeter.rs`.

    **Core types:**
    ```rust
    /// A single perimeter shell (one closed polygon path or set of paths).
    pub struct PerimeterShell {
        pub polygons: Vec<ValidPolygon>,  // The offset contour(s) for this shell
        pub is_outer: bool,               // True for the outermost (visible) shell
    }

    /// Perimeters for a single contour (outer boundary + its holes).
    pub struct ContourPerimeters {
        pub shells: Vec<PerimeterShell>,       // Ordered per print order (respects wall_order)
        pub inner_contour: Vec<ValidPolygon>,  // Innermost offset result = infill boundary
    }
    ```

    **Main function:** `generate_perimeters(contours: &[ValidPolygon], config: &PrintConfig) -> Vec<ContourPerimeters>`

    Algorithm:
    1. Separate contours into outer boundaries (CCW) and holes (CW).
    2. For each outer boundary (and its associated holes -- simplified: process each contour independently for Phase 3):
       a. Compute offset distances: first shell offset = `-half_width` (half nozzle width inward from slice contour). Subsequent shells offset = `-full_width` from the previous shell.
       b. Use `slicecore_geo::offset_polygon()` with `JoinType::Miter` for each offset level.
       c. If any offset produces empty result, stop (polygon collapsed -- no more shells fit).
       d. Track the innermost shell's result as the infill boundary.
    3. Apply wall ordering: if `config.wall_order == WallOrder::InnerFirst`, reverse the shell order so inner shells print first. Mark the outermost shell with `is_outer = true`.
    4. Use `mm_to_coord()` for all offset distance conversions.

    Implementation detail: Process all contours (outers and holes) together via `offset_polygons()` for proper interaction between adjacent boundaries. Pass all contours at a given offset level together.

    **Tests:**
    - A 20mm square with wall_count=2 produces 2 shells, each with >= 1 polygon.
    - A 20mm square with wall_count=2 and OuterFirst order: shells[0].is_outer == true.
    - A 20mm square with wall_count=2 and InnerFirst order: shells[1].is_outer == true (last one is outer).
    - inner_contour is smaller than the original contour.
    - wall_count=0 produces no shells but still computes inner_contour.
    - A very small polygon (2mm) with wall_count=5 stops early when polygon collapses.

  <action>
    Implement rectilinear infill pattern generation in `infill.rs`.

    **Core types:**
    ```rust
    /// A line segment in integer coordinate space representing one infill extrusion.
    pub struct InfillLine {
        pub start: IPoint2,
        pub end: IPoint2,
    }

    /// Result of infill generation for a single layer.
    pub struct LayerInfill {
        pub lines: Vec<InfillLine>,  // Infill extrusion segments
        pub is_solid: bool,          // True if this is a solid infill region (top/bottom)
    }
    ```

    **Main function:** `generate_rectilinear_infill(infill_region: &[ValidPolygon], density: f64, angle_degrees: f64, line_width: f64) -> Vec<InfillLine>`

    Algorithm:
    1. If density <= 0.0 or infill_region is empty, return empty vec.
    2. If density >= 1.0, treat as solid fill (spacing = line_width).
    3. Compute line spacing: `spacing_mm = line_width / density`. Convert to coord units.
    4. Compute bounding box of infill_region (iterate all points in all polygons, find min/max x and y).
    5. Generate parallel scan lines across the bounding box:
       - For angle_degrees == 0: horizontal lines at regular y intervals.
       - For angle_degrees == 90: vertical lines at regular x intervals.
       - For Phase 3: support 0 and 90 degrees only (alternate per layer for cross-hatching). More angles can be added later.
    6. Clip each scan line against the infill region:
       - For each scan line, find intersections with all polygon edges.
       - Sort intersections by position along the line.
       - Take pairs of intersections as entry/exit points (even-odd rule).
       - Each entry/exit pair becomes one InfillLine.
    7. The line-polygon clipping is implemented directly (scanline intersection algorithm) rather than using clipper2 boolean ops, because clipping open lines against closed polygons is more naturally done with ray-casting. This is ~50 lines of code:
       - For a horizontal line at y=Y, iterate all edges of all polygons.
       - For each edge (p1, p2) where min(p1.y, p2.y) <= Y <= max(p1.y, p2.y), compute x intersection.
       - Sort x intersections, pair them up (x[0]-x[1], x[2]-x[3], ...).
       - Each pair is an InfillLine segment.

    **Helper function:** `alternate_infill_angle(layer_index: usize) -> f64`
    - Returns 0.0 for even layers, 90.0 for odd layers. Creates cross-hatching pattern.

    **Tests:**
    - A 20mm square region at 20% density produces infill lines (count > 0).
    - A 20mm square at 0% density produces empty infill.
    - A 20mm square at 100% density produces solid fill (many closely-spaced lines).
    - Infill lines are all within the bounding box of the region.
    - Infill at 0 degrees produces horizontal lines; at 90 degrees produces vertical lines.
    - Empty infill region returns empty lines.
    - alternate_infill_angle returns correct angles for layer indices 0, 1, 2, 3.
  </action>
