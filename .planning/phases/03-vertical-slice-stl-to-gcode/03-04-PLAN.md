<task type="auto">
  <name>Task 1: Implement planner module (skirt, brim, retraction, temperature, fan)</name>
  <files>
    crates/slicecore-engine/src/planner.rs
    crates/slicecore-engine/src/lib.rs
  </files>
  <action>
    Implement the print planning module in `planner.rs`.

    **Skirt generation:** `generate_skirt(first_layer_contours: &[ValidPolygon], config: &PrintConfig) -> Vec<ValidPolygon>`
    1. Compute the convex hull of all first-layer outer contours (CCW polygons) using `slicecore_geo::convex_hull()`. Collect all points from all CCW contours into a flat Vec<IPoint2>, compute hull.
    2. Create a ValidPolygon from the hull. If hull validation fails (e.g., all contours are degenerate), return empty Vec.
    3. Offset the hull outward by `config.skirt_distance` mm using `offset_polygon()` with JoinType::Round.
    4. For additional skirt loops (config.skirt_loops > 1), offset outward from the previous skirt by one nozzle width per additional loop.
    5. Return all skirt loop polygons.

    **Brim generation:** `generate_brim(first_layer_contours: &[ValidPolygon], config: &PrintConfig) -> Vec<ValidPolygon>`
    1. If config.brim_width <= 0.0, return empty Vec.
    2. Compute number of brim loops: `brim_loops = (config.brim_width / config.nozzle_diameter).ceil() as u32`.
    3. For each loop, offset the outer contours outward by `nozzle_diameter * (i + 1)` using `offset_polygons()`.
    4. Return all brim loop polygons (ordered from inner to outer for printing).

    **Retraction planning:** `plan_retraction(travel_distance: f64, config: &PrintConfig) -> Option<RetractionMove>`
    ```rust
    pub struct RetractionMove {
        pub retract_length: f64,   // mm
        pub retract_speed: f64,    // mm/s
        pub z_hop: f64,            // mm (0.0 = no hop)
    }
    ```
    - Return `Some(RetractionMove)` if `travel_distance >= config.min_travel_for_retract`.
    - Return `None` for short travels (no retraction needed).

    **Temperature planning:** `plan_temperatures(layer_index: usize, config: &PrintConfig) -> Vec<GcodeCommand>`
    - Layer 0: Set first_layer_nozzle_temp and first_layer_bed_temp (with wait).
    - Layer 1: If first_layer temps differ from normal temps, emit temp change commands (no wait for bed, no wait for nozzle since it happens while printing).
    - Other layers: return empty Vec.

    **Fan control:** `plan_fan(layer_index: usize, layer_time_seconds: f64, config: &PrintConfig) -> Vec<GcodeCommand>`
    - If layer_index < config.disable_fan_first_layers: emit FanOff.
    - Else if layer_time_seconds < config.fan_below_layer_time: emit SetFanSpeed(config.fan_speed).
    - Else: emit SetFanSpeed at reduced speed proportional to how close layer_time is to fan_below_layer_time. For Phase 3 simplicity: just use full fan_speed whenever fan is enabled.

    **Tests:**
    - generate_skirt with a 20mm square contour and skirt_loops=1 produces 1 polygon larger than the input.
    - generate_skirt with skirt_loops=3 produces 3 polygons.
    - generate_brim with brim_width=0.0 returns empty.
    - generate_brim with brim_width=2.0 and nozzle_diameter=0.4 produces 5 brim loops.
    - plan_retraction with travel < min_travel_for_retract returns None.
    - plan_retraction with travel >= min_travel_for_retract returns Some with correct values.
    - plan_temperatures at layer 0 emits temp commands with wait.
    - plan_temperatures at layer 1 emits temp change (if different).
    - plan_fan at layer 0 (disable_fan_first_layers=1) emits FanOff.
    - plan_fan at layer 1 emits SetFanSpeed.
  </action>

<task type="auto">
  <name>Task 2: Implement G-code generation from toolpaths</name>
  <files>
    crates/slicecore-engine/src/gcode_gen.rs
    crates/slicecore-engine/src/lib.rs
  </files>
  <action>
    Implement G-code generation in `gcode_gen.rs`. This module converts the internal toolpath representation into GcodeCommand sequences.

    **Main function:** `generate_layer_gcode(toolpath: &LayerToolpath, config: &PrintConfig, retracted: &mut bool) -> Vec<GcodeCommand>`

    Algorithm:
    1. Emit layer comment: `; Layer {layer_index} at Z={z:.3}`.
    2. Emit Z-move (G0 to layer Z height) if this is a new layer.
    3. For each ToolpathSegment in the layer:
       a. If feature is Travel:
          - Check if retraction needed (plan_retraction based on segment length).
          - If retracting and not already retracted: emit Retract command, set `*retracted = true`.
          - If retracting and z_hop > 0: emit Z-hop (G0 Z+z_hop).
          - Emit RapidMove to the travel destination (G0 X Y F travel_speed*60).
          - If z_hop was applied: emit G0 back down to layer Z.
          - If retracted: emit Unretract, set `*retracted = false`.
       b. If feature is extrusion (perimeter, infill, skirt, brim):
          - If retracted from previous travel: emit Unretract first, set `*retracted = false`.
          - Emit LinearMove (G1 X Y E F).
          - E-value from the ToolpathSegment.
          - F (feedrate) from the ToolpathSegment (already in mm/min).
    4. Return the Vec<GcodeCommand>.

    **Full file generation:** `generate_full_gcode(layer_toolpaths: &[LayerToolpath], config: &PrintConfig) -> Vec<GcodeCommand>`

    1. Prepend: temperature commands, relative extrusion mode (M83), extruder reset (G92 E0).
    2. For each layer:
       a. Call plan_temperatures() for temp changes.
       b. Call plan_fan() for fan commands (using estimated_time_seconds from LayerToolpath).
       c. Call generate_layer_gcode().
    3. The start/end G-code is NOT generated here -- that's handled by GcodeWriter from slicecore-gcode-io. This function produces only the print body commands.
    4. Return all GcodeCommands for the full print.

    **Tests:**
    - generate_layer_gcode with a simple 2-segment extrusion layer produces G1 moves with E values.
    - Travel segments produce G0 moves.
    - Retraction inserts Retract/Unretract commands around long travels.
    - Short travels skip retraction.
    - Z-hop works when retract_z_hop > 0.
    - generate_full_gcode produces commands starting with M83 and G92 E0.
    - Feature type comments are included for readability.
  </action>

  </verify>

  </done>
