    Create comprehensive integration tests verifying each RENDER requirement.

    1. Create `crates/slicecore-render/tests/integration.rs`:

       **Helper: synthetic test mesh**
       - Build a simple cube (8 vertices, 12 triangles) programmatically using slicecore_mesh::TriangleMesh::new()
       - Build a simple pyramid (5 vertices, 6 triangles) for a different geometry

       **RENDER-01: Framebuffer creation and pixel write with z-test**
       - Create framebuffer, write pixel at (10, 10) with z=0.5
       - Write another pixel at (10, 10) with z=0.3 (closer) -- should overwrite
       - Write pixel at (10, 10) with z=0.8 (farther) -- should NOT overwrite
       - Assert final color matches z=0.3 pixel color

       **RENDER-02: Triangle rasterization produces correct pixels**
       - Render cube mesh from front angle at 100x100
       - Count non-background pixels -- should be >0 (cube visible from front)
       - Render from same angle twice -- pixel output is identical (deterministic)

       **RENDER-03: Camera angles produce distinct views**
       - Render cube from all 6 angles at 64x64
       - For each pair of angles, compare RGBA buffers -- at least some pixels must differ
       - All 6 must be pairwise distinct (total 15 pairs, all different)

       **RENDER-04: Gouraud shading varies with surface orientation**
       - Render cube from isometric -- cube has faces at different angles to light
       - Sample pixels from left region vs right region of the rendered image
       - Left and right should have different brightness (different face orientations to light)
       - Alternative: render a sphere-like mesh and verify brightness gradient exists

       **RENDER-05: PNG encoding produces valid PNG file**
       - Render cube to PNG via render_mesh
       - Assert png_data starts with PNG magic bytes: [0x89, 0x50, 0x4E, 0x47]
       - Assert png_data length > 100 bytes (non-trivial image)
       - Decode the PNG back using png crate to verify it's a valid PNG with correct dimensions

       **RENDER-06: 3MF export includes thumbnail attachment**
       - Build a TriangleMesh cube
       - Render thumbnail via render_mesh
       - Call save_mesh_with_thumbnail with the PNG bytes
       - Read the 3MF ZIP back, verify "Metadata/thumbnail.png" entry exists
       - Verify entry content matches the input PNG bytes

       **RENDER-07: G-code output includes thumbnail comment block**
       - Render thumbnail, format as G-code comment block via format_gcode_thumbnail_block
       - Verify output starts with "; thumbnail begin WxH SIZE"
       - Verify output ends with "; thumbnail end"
       - Verify all intermediate lines start with "; " and are <= 80 chars
       - Base64-decode the concatenated content (stripping prefixes) and verify it matches original PNG
       - Test both PrusaSlicer and Creality formats

       **RENDER-08: CLI thumbnail subcommand produces PNG file**
       - (In cli_thumbnail.rs) -- see below

       **RENDER-09: WASM compilation succeeds**
       - Not a runtime test -- verified by build step: `cargo build -p slicecore-render --target wasm32-unknown-unknown`
       - Add a comment-only test that documents this: `#[test] fn wasm_compilation_verified_by_ci() { /* See CI: cargo build --target wasm32-unknown-unknown */ }`

    2. Create `crates/slicecore-cli/tests/cli_thumbnail.rs`:
       - Use `std::process::Command` to invoke the built CLI binary
       - Create a temp dir with a synthetic STL file (write a minimal binary STL to disk)
       - Test: `slicecore thumbnail input.stl --output output.png --resolution 64x64`
         Assert output.png exists and starts with PNG magic bytes
       - Test: `slicecore thumbnail input.stl --angles front,back --resolution 64x64 --output {tmpdir}`
         Assert both input_front.png and input_back.png exist
       - Test: `slicecore thumbnail --help` exits 0 and output contains "thumbnail"

    3. Run full test suite:
