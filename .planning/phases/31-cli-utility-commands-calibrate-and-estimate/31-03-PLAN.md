<task type="auto">
  <name>Task 1: Temperature tower mesh generation and command</name>
  <files>crates/slicecore-engine/src/calibrate.rs, crates/slicecore-cli/src/calibrate/temp_tower.rs, crates/slicecore-cli/src/calibrate/mod.rs</files>
  <action>
    1. In calibrate.rs add `pub fn generate_temp_tower_mesh(params: &TempTowerParams) -> TriangleMesh`:
       - Generate a 1mm base plate at full width/depth using `primitive_box`
       - Stack N blocks (one per temperature step) of `block_height` each using primitive_box + translate + mesh_union
       - Number of blocks = ((end_temp - start_temp) / step).abs() as usize + 1
       - Each block positioned at z = 1.0 + i * block_height
       - Tower dimensions: base_width x base_depth x (1.0 + blocks * block_height)
       - Default dimensions: 30mm x 30mm, 8mm per block -- these fit most printers
       - Keep it simple -- stacked boxes, no bridging sections or embossed text (per CONTEXT.md: start simple)

    2. Create `crates/slicecore-cli/src/calibrate/temp_tower.rs`:
       - `pub fn cmd_temp_tower(args: TempTowerArgs) -> Result<(), Box<dyn std::error::Error>>` where TempTowerArgs contains all CLI fields from the TempTower variant
       - Pipeline: resolve config -> build TempTowerParams (profile defaults + CLI overrides) -> generate mesh -> validate_bed_fit -> optionally save_model -> slice via Engine -> inject_temp_changes -> write G-code with calibration header -> write companion instructions
       - Companion instructions (.instructions.md) explain: what each block tests, how to read results (look for stringing, layer adhesion, surface quality), which temperature to select
       - G-code header includes: test type, temperature range, step size, block height, SliceCore version
       - Output path defaults to `temp_tower.gcode` in current directory if not specified
       - Print summary to stderr: "Generated temperature tower: {start}C to {end}C in {step}C steps ({n} blocks)"

    3. Update calibrate/mod.rs: replace TempTower "not yet implemented" stub with call to `temp_tower::cmd_temp_tower()`. Add `mod temp_tower;`.

    Add unit test in calibrate.rs: `test_generate_temp_tower_mesh` verifies mesh has vertices, correct approximate height, and is non-empty.
  </action>

    <automated>cargo test -p slicecore-engine calibrate && cargo test -p slicecore-cli --lib && cargo build -p slicecore-cli 2>&1 | tail -5</automated>
  </verify>
  <done>temp-tower generates a mesh, slices it, injects temperature changes, and writes valid G-code + instruction file. Profile-aware defaults work.</done>
</task>

<task type="auto">
  <name>Task 2: Retraction test mesh generation and command</name>
  <files>crates/slicecore-engine/src/calibrate.rs, crates/slicecore-cli/src/calibrate/retraction.rs, crates/slicecore-cli/src/calibrate/mod.rs</files>
  <action>
    1. In calibrate.rs add `pub fn generate_retraction_mesh(params: &RetractionParams) -> TriangleMesh`:
       - Generate a multi-post model: 2 cylindrical posts connected by thin bridges at varying Z heights
       - Each bridge height corresponds to a retraction distance step
       - Use primitive_cylinder for posts (radius 5mm, total height = blocks * block_height + base)
       - Post spacing: 40mm center-to-center (gives ~30mm travel for retraction testing)
       - Use primitive_box for connecting bridges (thin bars, 2mm thick, 1mm tall)
       - Alternative simpler approach: single tower similar to temp tower with retraction distance changing per block. This is simpler and still effective -- use stacked boxes like temp tower, inject retraction distance overrides at Z boundaries
       - Number of steps = ((end_distance - start_distance) / step).abs() as usize + 1
       - Use the simpler tower approach (stacked boxes) for Phase 31 per CONTEXT.md guidance to keep it simple

    2. In calibrate.rs add `pub fn inject_retraction_changes(commands: Vec<GcodeCommand>, schedule: &[(f64, f64)]) -> Vec<GcodeCommand>`:
       - At Z height boundaries, inject G-code comments showing the current retraction distance
       - Insert M207 (firmware retraction set) commands if printer supports firmware retraction, otherwise inject retraction distance as a comment for reference since actual retraction is handled by the slicer
       - For simplicity in Phase 31: just add comments at Z boundaries indicating the retraction setting that was used for that section. The actual retraction distance variation comes from slicing each block with different retraction settings -- but since we slice the whole tower at once, we must post-process to document which section tests which distance. The user reads stringing quality per section
       - Note: True retraction variation per-block would require slicing each block separately or modifying retraction in post-process. For Phase 31, generate the tower with the profile's retraction settings and add documentation comments. The user manually adjusts retraction between prints for now

    3. Create `crates/slicecore-cli/src/calibrate/retraction.rs`:
       - `pub fn cmd_retraction(args: RetractionArgs) -> Result<(), Box<dyn std::error::Error>>`
       - Pipeline: resolve config -> build RetractionParams -> generate mesh -> validate_bed_fit -> optionally save_model -> slice -> add retraction documentation comments -> write G-code + instructions
       - Companion instructions explain: what each section tests, how to identify stringing/blobs, how to select optimal retraction distance
       - Summary to stderr: "Generated retraction test: {start}mm to {end}mm in {step}mm steps"

    4. Update calibrate/mod.rs: wire Retraction variant, add `mod retraction;`.

    Add unit test: `test_generate_retraction_mesh` verifies mesh is valid.
