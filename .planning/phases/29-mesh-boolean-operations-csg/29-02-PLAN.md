    2. Create `crates/slicecore-mesh/src/csg/intersect.rs`:

       Data types:
       - `IntersectionPoint { position: Point3, mesh_a_triangle: usize, mesh_b_triangle: usize, edge_param: f64 }` -- a point where meshes intersect
       - `IntersectionSegment { start: usize, end: usize, tri_a: usize, tri_b: usize }` -- indices into point registry, which triangle pair produced this segment
       - `IntersectionResult { points: Vec<IntersectionPoint>, segments: Vec<IntersectionSegment> }` -- all intersection data

       Main function:
       - `compute_intersection_curves(mesh_a: &TriangleMesh, mesh_b: &TriangleMesh) -> IntersectionResult`
         - Step 1: Build BVH for mesh B (mesh A's BVH is already lazy-built)
         - Step 2: For each triangle in mesh A, get its AABB, query mesh B's BVH for overlapping triangle AABBs
         - Step 3: For each candidate (tri_a, tri_b) pair, compute the exact intersection segment:
           - Classify each vertex of tri_a against the plane of tri_b using `perturbed_orient3d`
           - Classify each vertex of tri_b against the plane of tri_a using `perturbed_orient3d`
           - If all vertices of either triangle are on the same side, no intersection -> skip
           - Otherwise, compute the two endpoints of the intersection line segment by interpolating along triangle edges that cross the other triangle's plane
         - Step 4: Use a point registry (HashMap or similar) to canonicalize intersection points -- each unique intersection point (within a merge tolerance ~1e-10) gets a single index. This prevents T-junctions.
         - Step 5: Collect all segments into IntersectionResult

       Key constraints:
       - Use `robust::orient3d` (via perturbed_orient3d) for ALL point-plane orientation tests. NEVER use raw floating-point dot product comparisons for geometric decisions.
       - Compute each intersection point ONCE and reference by index (prevent T-junctions from inconsistent computation)
       - Use arena-style Vec for intersection points to avoid per-intersection allocation pressure

    3. Update `csg/mod.rs` to add `pub mod intersect; pub mod perturb;`

    Unit tests:
    - Two non-intersecting boxes -> empty IntersectionResult
    - Two overlapping boxes -> non-empty segments forming closed curves
    - Two identical boxes -> handled by perturbation (no crash)
    - Box and sphere partially overlapping -> intersection segments form a closed loop
    - Edge case: single shared edge between two triangles

    1. Create `crates/slicecore-mesh/src/csg/retriangulate.rs`:

       - `SplitTriangle { original_index: usize, mesh_id: MeshId, sub_triangles: Vec<[u32; 3]> }` -- result of splitting one triangle
       - `MeshId` enum: `A, B` -- which mesh the triangle came from
       - `retriangulate_mesh(vertices: &[Point3], indices: &[[u32; 3]], intersection_result: &IntersectionResult, mesh_id: MeshId, intersection_points: &[Point3]) -> (Vec<Point3>, Vec<[u32; 3]>, Vec<TriangleOrigin>)`
         - For each triangle in the mesh:
           - If it has no intersection segments crossing it, keep it as-is
           - If intersection segments cross it, split it using ear-clipping:
             a. Collect the intersection points that lie on this triangle's edges
             b. Insert these points as new vertices
             c. The intersection segments form constraints within the triangle
             d. Use ear-clipping to triangulate the resulting polygon(s)
             e. Each sub-triangle inherits the original triangle's winding order
         - Returns new vertices (original + intersection points), new indices, and a TriangleOrigin for each output triangle mapping back to source mesh and original triangle index
       - `TriangleOrigin { mesh_id: MeshId, original_triangle: usize }` -- provenance tracking

       Ear-clipping implementation:
       - Project polygon to 2D by dropping the coordinate with largest normal component
       - Find ears (vertices where the diagonal is inside the polygon and no other vertex is inside the ear triangle)
       - Clip ears iteratively until only a triangle remains
       - This is sufficient because splitting a triangle by line segments produces convex sub-polygons

    2. Create `crates/slicecore-mesh/src/csg/classify.rs`:

       - `Classification` enum: `Inside, Outside, Boundary`
       - `classify_triangles(mesh_vertices: &[Point3], mesh_indices: &[[u32; 3]], other_mesh: &TriangleMesh, triangle_origins: &[TriangleOrigin], intersection_result: &IntersectionResult) -> Vec<Classification>`
         - For each triangle in the (possibly retriangulated) mesh:
           - If the triangle was created by splitting (has intersection segments), it's on the boundary -> classify based on which side of the intersection curve it falls
           - If the triangle was NOT split:
             a. Compute centroid of the triangle
             b. Cast a ray from centroid in +X direction (or other axis if degenerate)
             c. Count intersections with the other mesh using BVH-accelerated ray casting
             d. Odd count -> INSIDE, even count -> OUTSIDE
             e. If ray hits exact vertex/edge (degenerate), perturb ray direction slightly and retry (up to 3 attempts with different axes)
       - Connected component optimization: all triangles in a connected component of non-intersected triangles share the same classification. Classify one triangle per component and propagate.

    3. Update `csg/mod.rs` to add `pub mod retriangulate; pub mod classify;`

    Unit tests for retriangulate:
    - Triangle with no intersections -> unchanged
    - Triangle split by one segment -> 2-3 sub-triangles, all valid, combined area equals original
    - Triangle split by two segments -> correct sub-triangle count

    Unit tests for classify:
    - Small box fully inside large box -> all triangles classified INSIDE
    - Small box fully outside large box -> all triangles classified OUTSIDE
    - Overlapping boxes -> correct mix of INSIDE and OUTSIDE
