```bash
#!/bin/bash
# Runs a single criterion benchmark with /usr/bin/time -v to capture peak RSS.
# Usage: ./scripts/bench-with-memory.sh <package> <bench-name> [extra-cargo-args...]
# Output: Appends bencher-format timing to bench-results/output.txt
#         Appends "MEMORY:<bench-name>:<peak_rss_kb>kB" to bench-results/memory.txt
set -euo pipefail

PACKAGE="$1"
BENCH_NAME="$2"
shift 2

mkdir -p bench-results

TIME_OUTPUT=$(mktemp)

# Run bench: stdout = bencher format timing, stderr = /usr/bin/time stats
/usr/bin/time -v cargo bench -p "$PACKAGE" --bench "$BENCH_NAME" "$@" \
  -- --output-format bencher 2>"$TIME_OUTPUT" | tee -a bench-results/output.txt

# Extract peak RSS from /usr/bin/time output
PEAK_RSS=$(grep "Maximum resident set size" "$TIME_OUTPUT" | awk '{print $NF}')
echo "MEMORY:${BENCH_NAME}:${PEAK_RSS}kB" >> bench-results/memory.txt

cat "$TIME_OUTPUT" >&2
rm -f "$TIME_OUTPUT"
```

Create `scripts/check-bench-regressions.sh` that enforces both timing and memory regression thresholds. This script is called by the CI bench job on PRs (when bench-ok label is NOT present).

```bash
#!/bin/bash
# Checks benchmark results for regressions exceeding thresholds.
# Parses criterion's bencher-format output for timing regressions.
# Parses bench-results/memory.txt for memory regressions against a baseline.
#
# Exit codes:
#   0 = no regressions above block threshold
#   1 = one or more regressions exceed 15% (block threshold)
#
# Emits GitHub Actions annotations:
#   ::warning:: for regressions between 5% and 15%
#   ::error::   for regressions above 15%
#
# Usage: ./scripts/check-bench-regressions.sh [base-memory-file]
#   base-memory-file: optional path to baseline memory.txt for memory comparison
set -euo pipefail

BLOCK_THRESHOLD=15
WARN_THRESHOLD=5
BLOCKED=0

echo "::group::Benchmark regression analysis"

# --- Timing regression check ---
# criterion-compare-action handles the detailed timing comparison and PR comment.
# This script provides the hard block enforcement by parsing criterion's output.
# Criterion outputs lines like:
#   test bench_name ... bench: 1234 ns/iter (+/- 56)
# When comparing with --baseline, it also outputs:
#   bench_name: ... regressed by 12.34%
# However, criterion-compare-action runs its own comparison internally.
# We parse its PR comment output or the action's exit behavior for timing.
#
# For timing: criterion-compare-action v3 posts annotations on regressions.
# We rely on criterion-compare-action for timing comparison accuracy since it
# runs both base and head on the same hardware.
#
# The real enforcement this script adds is for MEMORY regressions,
# which criterion-compare-action does not handle.

echo "Timing comparison: Handled by criterion-compare-action (same-hardware base vs head)."
echo ""

# --- Memory regression check ---
MEMORY_FILE="bench-results/memory.txt"
BASE_MEMORY_FILE="${1:-}"

if [[ -z "$BASE_MEMORY_FILE" || ! -f "$BASE_MEMORY_FILE" ]]; then
  echo "No baseline memory file provided or found. Skipping memory regression check."
  echo "(Memory baselines are established after the first main push.)"
  echo "::endgroup::"
  exit 0
fi

if [[ ! -f "$MEMORY_FILE" ]]; then
  echo "::error::bench-results/memory.txt not found. Benchmark memory tracking failed."
  echo "::endgroup::"
  exit 1
fi

echo "Comparing memory: $MEMORY_FILE vs baseline: $BASE_MEMORY_FILE"
echo ""

while IFS= read -r line; do
  # Parse MEMORY:<bench-name>:<peak_rss>kB
  if [[ "$line" =~ ^MEMORY:([^:]+):([0-9]+)kB$ ]]; then
    BENCH="${BASH_REMATCH[1]}"
    CURRENT="${BASH_REMATCH[2]}"

    # Find matching baseline
    BASE=$(grep "^MEMORY:${BENCH}:" "$BASE_MEMORY_FILE" | sed 's/.*:\([0-9]*\)kB/\1/' || echo "")

    if [[ -z "$BASE" || "$BASE" -eq 0 ]]; then
      echo "  $BENCH: ${CURRENT}kB (no baseline, skipping)"
      continue
    fi

    # Calculate percentage change: (current - base) / base * 100
    DELTA=$(( (CURRENT - BASE) * 100 / BASE ))

    if [[ "$DELTA" -gt "$BLOCK_THRESHOLD" ]]; then
      echo "::error::Memory regression: $BENCH regressed ${DELTA}% (${BASE}kB -> ${CURRENT}kB, threshold: ${BLOCK_THRESHOLD}%)"
      BLOCKED=1
    elif [[ "$DELTA" -gt "$WARN_THRESHOLD" ]]; then
      echo "::warning::Memory regression: $BENCH regressed ${DELTA}% (${BASE}kB -> ${CURRENT}kB)"
    elif [[ "$DELTA" -lt "-${WARN_THRESHOLD}" ]]; then
      echo "  $BENCH: improved ${DELTA}% (${BASE}kB -> ${CURRENT}kB)"
    else
      echo "  $BENCH: ${DELTA}% change (${BASE}kB -> ${CURRENT}kB) -- within tolerance"
    fi
  fi
done < "$MEMORY_FILE"

echo ""
echo "::endgroup::"

if [[ "$BLOCKED" -eq 1 ]]; then
  echo "::error::One or more benchmarks exceed the ${BLOCK_THRESHOLD}% memory regression threshold. Add the 'bench-ok' label to override."
  exit 1
fi

echo "All memory benchmarks within thresholds."
exit 0
```

Make the script executable with `chmod +x scripts/check-bench-regressions.sh`.

Key design decisions:
- Timing regression enforcement is delegated to criterion-compare-action (it runs both base and head on same hardware for accuracy). criterion-compare-action v3 posts annotations and comments automatically.
- Memory regression enforcement is this script's primary purpose since neither GitHub Action handles memory data.
- The script uses integer arithmetic (bash) for percentage calculation -- sufficient for 5%/15% thresholds.
- Emits GitHub Actions `::warning::` and `::error::` annotations for visibility.
- Exits non-zero (1) when any memory regression exceeds the block threshold.

**Job 2: `bench` -- benchmark job**

```yaml
  bench:
    name: Benchmarks
    needs: [changes]
    if: needs.changes.outputs.code == 'true'
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write
    steps:
      - uses: actions/checkout@v6
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
        with:
          prefix-key: bench

      # Run all 4 benchmark suites with memory tracking
      - name: Run benchmarks
        run: |
          chmod +x scripts/bench-with-memory.sh
          ./scripts/bench-with-memory.sh slicecore-engine slice_benchmark
          ./scripts/bench-with-memory.sh slicecore-engine geometry_benchmark
          ./scripts/bench-with-memory.sh slicecore-engine parallel_benchmark --features parallel
          ./scripts/bench-with-memory.sh slicecore-mesh csg_bench

      - name: Display memory results
        run: cat bench-results/memory.txt

      # PR comparison via criterion-compare-action (pull_request only)
      - name: Compare benchmarks (PR)
        if: github.event_name == 'pull_request'
        uses: boa-dev/criterion-compare-action@v3
        with:
          branchName: ${{ github.base_ref }}
          token: ${{ secrets.GITHUB_TOKEN }}

      # Check memory regression thresholds on PR (custom script)
      # Timing regressions are handled by criterion-compare-action (same-hardware comparison).
      # Memory regressions need custom enforcement since no action handles memory data.
      - name: Check regression thresholds
        if: >-
          github.event_name == 'pull_request' &&
          !contains(github.event.pull_request.labels.*.name, 'bench-ok')
        run: |
          chmod +x scripts/check-bench-regressions.sh
          ./scripts/check-bench-regressions.sh

      # Historical tracking (push to main only)
      # alert-threshold '105%' means warn when result is >=105% of baseline (i.e., >=5% slower).
      # This is the WARN tier only. The 15% BLOCK tier is enforced by check-bench-regressions.sh
      # on PRs (not on main pushes). RESEARCH.md example shows '115%' but that was for the
      # block tier; the historical tracker uses the warn tier since fail-on-alert is false.
      - name: Store benchmark result
        if: github.event_name == 'push' && github.ref == 'refs/heads/main'
        uses: benchmark-action/github-action-benchmark@v1
        with:
          name: libslic3r-rs Benchmarks
          tool: 'cargo'
          output-file-path: bench-results/output.txt
          github-token: ${{ secrets.GITHUB_TOKEN }}
          gh-pages-branch: gh-pages
          benchmark-data-dir-path: dev/bench
          auto-push: true
          alert-threshold: '105%'
          comment-on-alert: true
          fail-on-alert: false
          max-items-in-chart: 100
```

Key details:
- `changes` job uses `dorny/paths-filter@v3` to detect code-only changes in `crates/**`, `Cargo.toml`, `Cargo.lock`, `scripts/**`
- `bench` job depends on `changes` and only runs when `code == 'true'`
- Runs on ubuntu-latest only (consistent hardware for comparisons)
- Uses `Swatinem/rust-cache@v2` with `prefix-key: bench` to avoid cache collision with other jobs
- `criterion-compare-action@v3` runs on PRs for base-vs-head comparison with PR comment
- `github-action-benchmark@v1` runs on main pushes for historical tracking with `alert-threshold: '105%'` (5% warn tier). This is intentionally 105%, not 115%: the historical tracker uses `fail-on-alert: false` so it only warns. The 15% block is enforced by `check-bench-regressions.sh` on PRs via criterion-compare-action's same-hardware comparison
- Regression block step invokes `check-bench-regressions.sh` which exits non-zero on >15% memory regression
- The `bench-ok` label check is in the `if:` condition -- when the label is present, the regression check step is skipped entirely
- `permissions: contents: write` for gh-pages push, `pull-requests: write` for PR comments
- Existing jobs (fmt, clippy, test, test-linux-arm, wasm, doc) are UNCHANGED
