  <name>Task 1: Extract shared process_single_layer helper and convert slice_to_writer_with_events to parallel</name>
  <files>crates/slicecore-engine/src/engine.rs</files>
  <action>
This is the main parallelization task. The approach:

**Step 1: Initialize thread pool at engine construction or slice entry.**
At the start of `slice_to_writer_with_events` (and the other slice methods), call `crate::parallel::init_thread_pool(self.config.thread_count)` to configure rayon's global pool if the user specified a thread count.

**Step 2: Determine parallel mode at runtime.**
Create a local boolean `use_parallel`:
```rust
let use_parallel = self.config.parallel_slicing
    && self.config.infill_pattern != InfillPattern::Lightning;
```
When infill pattern is Lightning, force sequential mode (lightning builds cross-layer tree state). When `parallel` feature is not enabled, `maybe_par_iter!` compiles to `iter()` anyway, so the two-pass logic still works correctly but runs sequentially.

**Step 3: Convert the main layer loop in `slice_to_writer_with_events`.**
Replace the `for (layer_idx, layer) in layers.iter().enumerate()` loop (starting around line 869) with:

a) **Parallel path (when use_parallel is true):**
   - Use `maybe_par_iter!(layers).enumerate().map(|(layer_idx, layer)| { ... }).collect::<Result<Vec<_>, EngineError>>()?` to process all layers in parallel.
   - Inside the closure, perform steps 2a through 2i (polyhole, perimeters, surface classification, infill, gap fill, toolpath assembly, arachne, support, bridge, ironing) exactly as in the current sequential code.
   - Pass `None` for `previous_seam` in `assemble_layer_toolpath` (seam alignment deferred to pass 2).
   - Check cancellation at the start of each layer's closure: `if cancel_token.as_ref().map_or(false, |t| t.is_cancelled()) { return Err(EngineError::Cancelled); }`
   - Each closure returns `Ok((LayerToolpath, Option<IPoint2>))`.
   - The closure captures `&self.config`, `&layers`, `&lightning_ctx`, `&support_result`, `&cancel` -- all shared references, no mutation needed inside the closure.
   - NOTE: `self.generate_infill_for_layer()` takes `&self` which is fine for shared access in rayon closures (Engine is not mutated during slicing). However, if the borrow checker complains about `&self` in a rayon closure, extract the needed config fields into local variables before the parallel section.

b) **After parallel collect, apply two-pass seam adjustment:**
   ```rust
   let mut previous_seam: Option<IPoint2> = None;
   let layer_toolpaths: Vec<LayerToolpath> = layer_results.into_iter()
       .map(|(mut tp, seam)| {
           // For Aligned seam, adjust seam position using previous layer's seam.
           // This produces identical results to the sequential single-pass approach
           // because assemble_layer_toolpath with previous_seam=None still places
           // the seam, and the adjustment rotates the start point to align.
           // NOTE: If assemble_layer_toolpath's previous_seam logic is more complex
           // than a simple rotation, you may need to re-call the seam placement
           // function here. Inspect the actual assemble_layer_toolpath code to
           // determine the correct approach.
           if seam.is_some() { previous_seam = seam; }
           tp
       })
       .collect();
   ```
   IMPORTANT: Carefully inspect `assemble_layer_toolpath` to understand exactly what `previous_seam` does. If it influences the seam placement point selection (not just a post-hoc adjustment), then the two-pass approach needs to re-run the seam selection for each layer sequentially in pass 2, using the collected perimeter data. The key constraint is: parallel output must be bit-identical to sequential output.

c) **Sequential path (when use_parallel is false):**
   Keep the existing sequential loop logic exactly as-is, including per-layer event emission, `previous_seam` tracking, and `layer_durations` collection.

**Step 4: Handle progress events in parallel mode.**
In parallel mode, do NOT emit `LayerComplete` events per-layer (they'd arrive out of order). Instead:
- Before the parallel section, emit `StageChanged("layer_processing", 0.1)`.
- After the parallel collect completes, emit a summary progress event.
- Use `AtomicProgress` from parallel.rs if you want mid-parallel progress (increment inside closure, but don't emit events -- just track the counter).

**Step 5: Apply the same conversion to `slice_with_preview` and `slice_with_modifiers`.**
Both methods have nearly identical layer loops (lines ~1522 and ~1801). Apply the same parallel/sequential branching pattern. If the code duplication is excessive (it likely will be), consider extracting a helper function `process_layers_parallel(...)` that encapsulates the parallel/sequential dispatch, used by all three methods.

**Step 6: Handle the `layers` borrow in classify_surfaces.**
`classify_surfaces(&layers, layer_idx, ...)` borrows the full `layers` slice. This is fine for parallel closures -- it's a shared `&[SliceLayer]` reference. But ensure the closure captures `&layers` (not `layers`).

**Critical implementation detail for bit-identical output:**
The `previous_seam` parameter is used by `assemble_layer_toolpath` to influence seam placement for `SeamPosition::Aligned`. Read the function to understand exactly what it does:
- If `previous_seam` only affects the starting point of the outer perimeter (rotating the polygon start), then pass `None` in parallel mode and apply the rotation in pass 2.
- If `previous_seam` affects which corner/edge is selected as seam point, then the two-pass approach requires re-running seam selection sequentially. In this case, collect the perimeter data from pass 1 and re-run `assemble_layer_toolpath` or its seam-specific subroutine in pass 2.
- The safest approach: in parallel mode, compute everything EXCEPT seam placement in pass 1, then do full seam placement sequentially in pass 2.
