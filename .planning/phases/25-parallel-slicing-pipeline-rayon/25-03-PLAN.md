  <action>
1. Create `crates/slicecore-engine/benches/parallel_benchmark.rs`:

   - Use the same synthetic mesh construction pattern as `slice_benchmark.rs` (look at how it creates test meshes -- likely a calibration cube or sphere).
   - Create a test mesh with enough layers to benefit from parallelism. A taller mesh is better: e.g., a 40mm tall cube at 0.2mm layer height = 200 layers. Use the existing synthetic mesh builder or create a simple box mesh programmatically.
   - Create a benchmark group "parallel_vs_sequential" with two benchmarks:
     a) `sequential`: PrintConfig with `parallel_slicing: false`, slice the mesh
     b) `parallel`: PrintConfig with `parallel_slicing: true` (default), slice the mesh
   - Use `criterion::BenchmarkGroup` with `sample_size(10)` (slicing is slow, don't need many samples).
   - Use `criterion::measurement::WallTime` (default).
   - The benchmark should measure the full `engine.slice()` call including G-code generation (since we only parallelize layer processing, the total speedup will be a fraction of the layer processing time).
   - Add a third benchmark variant if useful: `parallel_4_threads` with `thread_count: Some(4)` to show scaling behavior.

2. Add the `[[bench]]` entry to `Cargo.toml`:
   ```toml
   [[bench]]
   name = "parallel_benchmark"
   harness = false
   required-features = ["parallel"]
   ```
   The `required-features` ensures this benchmark only builds when the parallel feature is enabled.

3. Verify the benchmark runs with: `cargo bench -p slicecore-engine --bench parallel_benchmark --features parallel -- --quick`
   (The `--quick` flag makes criterion run fewer iterations for a fast check.)
